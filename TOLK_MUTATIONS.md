# Мутации Tolk

В этом файле описаны Tolk-специфичные правила мутаций, добавленные в этом форке.

## Файлы правил

- Regex-правила: `universalmutator/static/tolk.rules`
- Comby-правила: `universalmutator/comby/tolk.rules`

При генерации мутантов также применяется `universal.rules`, поэтому к файлам
`.tolk` по-прежнему применяются арифметические, сравнительные, булевы,
числовые и другие языконезависимые мутации.

## Использованные источники

Tolk-специфичные правила были выведены из встроенных TON-справочников,
которые используются skill `ton-languages-regex-mutator`:

- `tolk/basic-syntax.mdx`
- `tolk/syntax/functions-methods.mdx`
- `tolk/features/message-handling.mdx`
- `tolk/syntax/structures-fields.mdx`

## Regex-движок

- `static/tolk.rules` использует синтаксис Python `re`.
- `comby/tolk.rules` использует шаблоны Comby.

Правила по возможности якорятся на началах строк или на точных ключевых
словах. Они изменяют только поверхностный синтаксис и не пытаются
переписывать вложенные блоки.

## Краткая таблица замен

| Что было                            | Во что мутирует            | Чем отличается                                                               |
| ----------------------------------- | -------------------------- | ---------------------------------------------------------------------------- |
| `true`                              | `false`                    | Инвертируется булев литерал.                                                 |
| `false`                             | `true`                     | Инвертируется булев литерал.                                                 |
| `val`                               | `var`                      | Неизменяемая локальная переменная становится изменяемой.                     |
| `is`                                | `!is`                      | Проверка типа по union/nullable меняется на отрицательную.                   |
| `!is`                               | `is`                       | Отрицательная проверка типа меняется на положительную.                       |
| `lazy X`                            | `X`                        | Ленивое вычисление заменяется на eager-вычисление.                           |
| `get fun name(...)`                 | `fun name(...)`            | Getter перестаёт быть getter-методом и становится обычной функцией.          |
| `mutate self`                       | `self`                     | У receiver убирается право на изменение `self`.                              |
| `BounceMode.NoBounce`               | `BounceMode.RichBounce`    | Сообщение вместо запрета bounce начинает использовать rich bounce.           |
| `BounceMode.RichBounce`             | `BounceMode.NoBounce`      | Rich bounce отключается.                                                     |
| `BounceMode.Only256BitsOfBody`      | `BounceMode.RichBounce`    | Короткий bounced-body заменяется на полный rich bounce-режим.                |
| `BounceMode.RichBounceOnlyRootCell` | `BounceMode.RichBounce`    | Ограниченный rich bounce заменяется на полный rich bounce.                   |
| `InMessageBounced`                  | `InMessage`                | Handler/параметр начинает ожидать обычное входящее сообщение вместо bounced. |
| `InMessage`                         | `InMessageBounced`         | Handler/параметр начинает ожидать bounced-сообщение вместо обычного.         |
| `struct (0x12345678) Name`          | `struct (0x00000000) Name` | Opcode/prefix структуры заменяется на нулевой.                               |
| `struct (0x12345678) Name`          | `struct (0xFFFFFFFF) Name` | Opcode/prefix структуры заменяется на максимальный 32-битный.                |

## Описанные Tolk-специфичные мутации

### Правила безопасности и пропуска

- `^\\s*import\\s ==> DO_NOT_MUTATE`
  Запрещает мутировать строки `import`.
- `// ==> SKIP_MUTATING_REST`
  Запрещает мутации в оставшейся части однострочного комментария.
- `/\\* ==> SKIP_MUTATING_REST`
  Запрещает regex-мутации после начала блочного комментария на той же строке.

### Булевы литералы и локальная изменяемость

- `true ==> false`
- `false ==> true`
- `\\bval\\b ==> var`

Это простые поверхностные мутации, соответствующие документированному
синтаксису Tolk для булевых литералов и локальных объявлений переменных.

### Проверки union-типов

- `(?<![!\\w])is\\b ==> !is`
- `(?<!\\w)!is\\b ==> is`

Эти правила нацелены на Tolk-проверки типов во время выполнения для union- и
nullable-значений. Они опираются на точные операторы `is` и `!is`,
задокументированные в языке.

Риск:

- Может получаться невалидный или семантически странный код, если проект
  использует необычные пробелы или похожие фрагменты текста в строках,
  переживающих фильтрацию комментариев.

### Lazy loading

- `(^|\\s)lazy(\\s+) ==> \\1`

Эта мутация убирает `lazy` и принудительно делает вычисление eager в
конструкциях вроде `lazy AllowedMessage.fromSlice(...)` или
`lazy Storage.load()`.

Риск:

- Меняется стратегия вычисления, расход газа и поведение со стеком.

### Объявления getter-функций

- `\\bget\\s+fun\\b ==> fun`

Эта мутация убирает getter-экспорт, сохраняя тело объявления функции.

Риск:

- Меняет публичное интерфейсное поведение контракта.

### Объявления изменяемого receiver

- `\\bmutate\\s+self\\b ==> self`

Эта мутация убирает изменяемость receiver в instance-методах, объявленных с
`mutate self`.

Риск:

- Часто превращает валидный мутирующий метод в невалидный код, если тело
  записывает в `self`.

### Мутации BounceMode

- `\\bBounceMode\\.NoBounce\\b ==> BounceMode.RichBounce`
- `\\bBounceMode\\.RichBounce\\b ==> BounceMode.NoBounce`
- `\\bBounceMode\\.Only256BitsOfBody\\b ==> BounceMode.RichBounce`
- `\\bBounceMode\\.RichBounceOnlyRootCell\\b ==> BounceMode.RichBounce`

Эти правила нацелены на задокументированные значения enum `BounceMode`,
используемые при отправке сообщений.

Риск:

- Высокий. Меняется семантика bounce, поведение по газу и форма данных,
  доступных в bounced-handler'ах.

### Зарезервированные типы message wrapper

- `\\bInMessageBounced\\b ==> InMessage`
- `\\bInMessage\\b ==> InMessageBounced`

Эти мутации затрагивают задокументированные типы параметров зарезервированных
message-handler'ов.

Риск:

- Высокий. Это может сломать сигнатуры handler'ов или изменить набор полей,
  доступных у параметра.

### Мутации opcode / prefix у struct

- `(^\\s*struct\\s+\\()0x[0-9A-Fa-f]{8}(\\)\\s+[A-Za-z_]\\w*) ==> \\g<1>0x00000000\\2`
- `(^\\s*struct\\s+\\()0x[0-9A-Fa-f]{8}(\\)\\s+[A-Za-z_]\\w*) ==> \\g<1>0xFFFFFFFF\\2`

Эти правила нацелены на задокументированные объявления вида
`struct (0x12345678) Name { ... }`, используемые для message-opcode и
serialization-prefix.

Риск:

- Очень высокий. Мутации prefix/opcode могут ломать ABI-совместимость,
  маршрутизацию, lazy-dispatch по union и предположения о layout
  сериализации.

## Покрытие Comby

`universalmutator/comby/tolk.rules` по возможности отражает тот же набор идей
через Comby:

- `val` в `var`
- `is` в `!is`
- `!is` в `is`
- удаление `lazy`
- `get fun` в `fun`
- `mutate self` в `self`
- выбранные rewrite для `BounceMode`
- `InMessageBounced` в `InMessage`
- `struct (PREFIX) Name {` в фиксированные prefix-значения

Набор Comby немного консервативнее regex-набора, потому что он старается
сохранять простое структурное сопоставление и не переусложнять модель
грамматики Tolk.

## Текущая философия

Набор Tolk-правил намеренно сделан консервативным:

- предпочтение точным документированным ключевым словам, а не широким
  переписываниям токенов;
- предпочтение мутациям в заголовках объявлений, а не переписыванию блоков;
- строки `import` остаются нетронутыми;
- мутации ABI, opcode и формы сообщений считаются высокорисковыми, но
  полезными;
- regex не должен притворяться безопасным парсером вложенного синтаксиса
  Tolk.
